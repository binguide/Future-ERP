# خطة تنفيذ ERPFuture بوكيل برمجة (نموذج متوسط القدرة)

> هذه الخطة موجَّهة لـ **وكيل برمجة آلي** ينفّذ المشروع مهمةً مهمة. صُمِّمت بافتراض أن النموذج
> المنفِّذ **متوسط القدرة** — لذا المهام صغيرة، صريحة، ولكل واحدة **معيار قبول قابل للتنفيذ**.
> المصدر المرجعي للنطاق والمعمارية هو [PRD.md](PRD.md)، والقواعد الحاكمة في [../CLAUDE.md](../CLAUDE.md).

---

## الجزء أ — نموذج تشغيل الوكيل (اقرأه قبل أي مهمة)

### مبادئ العمل
1. **مهمة واحدة في كل مرة.** لا تبدأ مهمة قبل اكتمال اعتمادياتها (`يعتمد على`).
2. **حجم المهمة = طلب دمج (PR) واحد صغير** قابل للمراجعة، مع اختباراته.
3. **TDD إلزامي لكل منطق مالي/مخزني** (الترحيل، الإلغاء، التقييم، حلّ الهرمية): اكتب الاختبار
   أولاً بأرقام معروفة النتيجة، ثم نفِّذ حتى ينجح.
4. **«تمّت» تعني**: تجتاز الاختبارات + `lint` + `typecheck` نظيفة + معيار القبول محقّق فعليًا
   (نفِّذ الفحص، لا تفترضه).
5. **توقّف واسأل** إذا كانت المهمة غامضة، أو تتطلّب تغيير أحد الـ invariants، أو تتطلّب اعتمادية مفقودة.
6. **لا تُدخل اعتمادية مدفوعة** (القائمة كلها MIT/OSS — راجع PRD ملحق أ).

### قالب المهمة (التزم به حرفيًا)
```
المعرّف:            T<phase>.<n>
يعتمد على:          [قائمة المعرّفات]
الهدف:              جملة واحدة
اقرأ أولًا:          ملفات/أقسام PRD ذات الصلة
الخطوات:            خطوات مرقّمة قصيرة
ملفات تُنشأ/تُعدَّل:  مسارات محدّدة
معيار القبول:        اختبارات/فحوص قابلة للتنفيذ (هذا هو تعريف النجاح)
الحدود الحمراء:      الـ invariants التي يجب ألا تُكسَر في هذه المهمة
```
> في القائمة أدناه يُذكر لكل مهمة **الاعتمادية + الهدف + معيار القبول** باختصار؛ على الوكيل توسيعها
> بالقالب الكامل قبل التنفيذ.

### قائمة الفحص قبل كل تسليم (الحدود الحمراء — من CLAUDE.md)
- [ ] لا أرقام مالية خارج `GLEntry`؛ كل ترحيل **متوازن بالعملة الأساسية** وداخل **معاملة DB واحدة**.
- [ ] العزل بين المستأجرين سليم (schema-per-tenant + `search_path`)؛ **لا** عمود `tenant_id`.
- [ ] دورة المستند سليمة: `Save` بلا أثر، `Submit` يولّد القيود ويقفل، `Cancel` يعكس؛ المُرحَّل لا يُحذف.
- [ ] إذا كان Workflow مفعّلًا: لا `Submit` قبل حالة «معتمَد».
- [ ] حلّ الهرمية يستخدم الـ resolver الموحَّد (الأخص يفوز)، لا منطقًا محليًا جديدًا.
- [ ] الأصناف الخدمية: لا حركة مخزون ولا قيد مخزون/تكلفة.
- [ ] كل قائمة/تقرير يحترم تصفية صلاحيات الصف (شركة/فرع/مخزن).
- [ ] السجل (audit/DocVersion) append-only.

### بوّابات تحقّق بين المراحل
لا تنتقل إلى المرحلة التالية قبل اجتياز **بوّابة المرحلة** (🚦 في نهاية كل مرحلة): مجموعة اختبارات/فحوص
تُثبت سلامة أساسات المرحلة. هذه نقاط مراجعة بشرية مقترحة.

- **T2.12 — ValuationStrategy: MovingAverage.** يعتمد: T2.7. يخزّن القيمة والمتوسط الجاريين. القبول: متوسط صحيح بعد استلامات بأسعار مختلفة.
- **T2.13 — ValuationStrategy: FIFO.** يعتمد: T2.7. طابور طبقات تكلفة لكل (صنف×مخزن). القبول: استهلاك بترتيب الورود صحيح.
- **T2.14 — مسار SerialNo.** يعتمد: T2.6, T2.13. تقييم بالتكلفة الفعلية للرقم. القبول: تكلفة البيع = تكلفة الرقم المستلَم.
- **T2.15 — ترحيل حركة المخزون للـ GL.** يعتمد: T2.9, T0.21. عبر `PostingService` (مخزون دائم)؛ **الخدمي يتخطّى**. القبول: حركة صنف خدمي لا تولّد قيد مخزون/تكلفة.
- **T2.16 — قيد الاستلام.** يعتمد: T2.15. Dr مخزون / Cr بضاعة مستلمة غير مفوترة. القبول: قيد متوازن صحيح القيمة.
- **T2.17 — قيد COGS عند البيع.** يعتمد: T2.15. Dr COGS / Cr مخزون بالتكلفة. القبول: التكلفة من ValuationStrategy الصحيحة.
- **T2.18 — العمليات المخزنية.** يعتمد: T2.16, T2.17. إذن استلام/صرف/تحويل + `StockReconciliation`. القبول: كل عملية تُولّد حركة وقيدًا صحيحين.
- **T2.19 — قفل وتغيير طريقة التقييم.** يعتمد: T2.10, T2.18. قفل بعد أول حركة؛ التغيير عبر reconciliation أو صلاحية `change_valuation_method` مع **إعادة احتساب آلية** من تاريخ القطع. القبول: تغيير بعد حركات يُطلق إعادة احتساب لا تبديلًا صامتًا.
- **T2.20 — معالجة الرصيد السالب.** يعتمد: T2.11, T2.13. رفض عند الإيقاف؛ آخر سعر معروف عند التفعيل + تسوية لاحقة. القبول: الرفض يعمل؛ والتسوية تصحّح القيود عند ورود الكمية.
- **T2.21 — إعادة احتساب FIFO عند الإلغاء/الأثر الرجعي.** يعتمد: T2.13, T0.24. القبول: إلغاء/تعديل بأثر رجعي يعيد بناء الطابور والقيود صحيحًا.
- **T2.22 — تقارير المخزون.** يعتمد: T0.34, T2.18. رصيد، حركة، أعمار، تقييم، حد إعادة طلب، أرصدة دفعة/رقم تسلسلي، قرب انتهاء، فروقات جرد. القبول: التقييم يطابق دفتر المخزون.
- 🚦 **بوّابة ٢**: صنفان بطريقتين مختلفتين تُحتسب تكلفتهما صحيحًا؛ رفض السالب عند الإيقاف؛ تسوية التكلفة عند التفعيل (PRD قسم ١٠).

## الجزء ج — تخفيف مخاطر «النموذج المتوسط»

- **اختبارات ذهبية (Golden tests)**: لكل مسار ترحيل/إلغاء سيناريو بأرقام ثابتة معروفة النتيجة؛ يمنع الانجراف الحسابي.
- **منع إعادة الاختراع**: استدعِ `PostingService` والـ resolver الموحَّد دائمًا؛ يُمنع منطق ترحيل/حلّ هرمية محلي جديد (يُرفض في المراجعة).
- **مهام صغيرة قابلة للتراجع**: كل مهمة في فرع/PR مستقل يُراجَع مقابل قائمة الفحص.
- **قائمة «ممنوع»**: لا حذف مستند مُرحَّل، لا أرقام مالية خارج GL، لا `tenant_id`، لا تجاوز Workflow، لا اعتماديات مدفوعة.
- **التوقّف عند الغموض** أفضل من التخمين: إن لم يكن معيار القبول قابلًا للتنفيذ، تُعاد صياغة المهمة قبل البدء.
